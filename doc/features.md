# Модули приложения

Краткое описание каждого модуля и его ключевых бизнес-правил. Полные приёмочные критерии —
[specs/001-timefocus-mvp/spec.md](../specs/001-timefocus-mvp/spec.md).

## Трекер активностей (`features/tracker`)

Сетка иконок активностей → тап запускает трекинг, карточка с тикающим таймером появляется
вверху списка. Пауза/возобновление/стоп. Несколько обычных активностей могут идти
параллельно. Активности-группы разворачиваются in-place (кнопка «Назад» вместо перехода на
отдельный экран).

Матрица переходов при старте новой активности определена в **одном месте** —
`StartActionUseCase`:

- активность с `mode == pomodoro` или `mode == breakFor`, либо флагом `pauseOthers` —
  прерывает текущий Помодоро-интервал (`interrupted`) и ставит остальные активности на паузу;
- системное завершение рабочего интервала Помодоро **не** прерывает сам Помодоро;
- второй запуск Помодоро при уже активном требует подтверждения пользователя.

## Помодоро (`features/pomodoro`)

Циклы работа → перерыв, 3 типа перерывов (short/long по номеру цикла в серии), настраиваемое
число циклов (3–5), действие после серии (`afterAction`, 5 вариантов). Интервал засчитывается
как `completed` только если завершился системно; `interrupted` не сбрасывает номер цикла —
только явный «Стоп» активности.

## Вода и HUD (`features/water`)

Всегда видимая HUD-панель (в shell, поверх всех вкладок): полоска выпитого/цели с меткой
ожидаемого объёма к текущему моменту, кнопка-стакан и **очередь контекстных
иконок-напоминаний**.

### Кнопка-стакан (`_GlassButton`)

Показывает иконку **последнего выбранного QuickButton** (не статичную `local_drink`), тап
логирует именно его объём (`cubit.logDrink`); long-press открывает список напитков — выбор там
и логирует, и запоминает выбор для следующих тапов. Выбор не персистится: при каждом
пересоздании виджета (перезапуск приложения) снова используется первый доступный QuickButton;
если активных QuickButton нет вообще — дефолтная иконка `Icons.local_drink_outlined` и лог
`AppConstants.defaultWaterPortionMl`. Пульсация (`glassBlinking`, Помодоро активен) — это
`AnimatedScale` поверх текущей иконки, а не переключение на другую иконку.

### Очередь контекстных иконок

### Очередь контекстных иконок

Это не отражение того, что сейчас запущено, а список подсказок вида «сходи сделай X» —
`HudCubit` + `HudQueueRepository`, персистентно хранится в таблице `HudQueueItems`
(`shared/database/tables/hud_tables.dart`). Ключ уникальности — сама активность
(`SystemAction`): у одной активности не может быть двух записей одновременно. Очередь
скоупится на календарный день — при каждом `subscribe()` (запуск/резюм приложения) всё, что
помечено не сегодняшним днём, удаляется (`purgeStale`). Т.е. данные переживают перезапуск
приложения в течение дня, но не накапливаются как многодневная история.

`HudQueueRepository` даёт два разных способа положить действие в очередь — разница принципиальна:

- **`raise`** (`upsert` в БД) — «оживляет» запись, снимает `dismissed`, даже если она уже была
  снята пользователем. Для источников, где каждый вызов — это реально новый повод (выпил воды
  ещё раз, начался ещё один перерыв).
- **`raiseIfNew`** (`insertIfAbsent`, `InsertMode.insertOrIgnore`) — только вставляет, если
  записи ещё нет вообще; уже существующую (в том числе снятую) не трогает. Для
  уровневых проверок вида «время уже наступило?», которые перезапускаются каждую минуту
  тикером и заново при каждом перезапуске приложения — без этого они бы «воскрешали» подсказку,
  которую пользователь только что снял или запустил (был реальный баг, исправлено).

Три источника, которые кладут иконку в очередь:

1. **Время по расписанию** (`features/schedule`) — любое включённое `ScheduleEvent`, чей
   `ScheduleEventType` соответствует какому-то `SystemAction` (любой тип, кроме `wakeUp` и
   `custom` — см. `ScheduleEventType.systemAction`), при наступлении своего времени добавляет
   эту активность в очередь через `raiseIfNew`. Проверка — раз в минуту таймером
   (`HudCubit._checkScheduledTriggers`) и один раз сразу при `subscribe()`.
2. **Питьё воды** — после `logWater`, если включена настройка `showToiletOnWater`, в очередь
   через `raise` добавляется `SystemAction.toilet` (плюс отдельно планируется
   push-напоминание).
3. **Начало перерыва Помодоро** — `HudCubit.onPomodoroBreakStarted()`, если включена
   настройка `showToiletOnBreak` — аналогично через `raise` добавляет `toilet`.

Важно: запуск активности вручную (из сетки трекера или тапом по самой HUD-иконке) **не**
создаёт запись в очереди — очередь только про напоминания, никогда про текущее состояние.

Сортировка — по `SystemAction.priority` (порядок объявления enum в `system_actions.dart`, в
БД не хранится), по убыванию: `medicine > toilet > warmup > meal > sport > prayer >
meditation > walk > breakFor > rest > work > sleep`.

UI (`HudPanel` / `HudContextIcon`): горизонтальный ряд иконок. Свайп вверх (`Dismissible`) —
снять подсказку без действия. Тап — запускает соответствующую системную активность
(`ActionEvent.startSystemAction`) и одновременно снимает иконку из очереди
(`HudCubit.onQueueItemTapped`).

### Push-напоминания о воде

Два режима, не связаны с очередью HUD-иконок:
- **interval** — следующее уведомление = `lastDrankAt + N минут`, пересоздаётся после каждого
  питья;
- **scheduled** — все уведомления дня планируются заранее, питьё не сбрасывает расписание;
  автопропуск в окне ±15 минут вокруг приёмов пищи, отбрасывание после окна сна.

## Расписание дня (`features/schedule`)

Вертикальный таймлайн на день с дорожками для параллельных интервалов: план (события),
факт (запуски активностей), вода, напоминания. События двух типов:

- **строгие** (`mealStrict` и т. п.) — если до начала не хватает времени на текущий
  Помодоро-интервал, предупреждение приходит немедленно; в точное время события Помодоро
  принудительно прерывается, активность стартует только по тапу на уведомление;
- **гибкие** — ждут завершения текущего Помодоро-интервала, пульсация HUD вместо прерывания.

`ScheduleEventType` хранится в БД по имени (не по индексу — можно свободно добавлять/
переставлять типы) и включает в себя все `SystemAction`, а не только `meal`/`sleep`/`wakeUp`:
любое событие расписания, чей тип соответствует системной активности, при наступлении своего
времени также попадает в очередь HUD-иконок (см. «Вода и HUD» выше).

## Уведомления (`features/notifications`)

Единая система на 9 типов (`NotificationType`), выше остальных модулей не имеет прямых
зависимостей — координация через `RootBlocListener`. Payload содержит всё необходимое для
разбора при холодном старте приложения (`{'actionId', 'breakActionId', 'pomodoroCount', ...}`).
Отложенные уведомления (Помодоро активен, мьют) копятся и доставляются все по очереди после
снятия блокировки. Разрешения зависят от версии Android (`POST_NOTIFICATIONS` с SDK 33,
`SCHEDULE_EXACT_ALARM` с SDK 31, с fallback на неточное планирование).

## История и отчёты (`features/history`)

Режимы день/неделя/месяц/произвольный период с навигацией стрелками и «сегодня». Шапка:
суммарное время (без активности «Сон»), статус Помодоро (завершено/прервано), вода
(выпито/цель). Редактирование интервала — быстрые кнопки сдвига времени, инлайн-валидация
(конец раньше начала — блокирует сохранение), пересечение интервалов одной активности —
предупреждает, но не блокирует. Экран отчётов — 7 пресетов периодов + графики (`fl_chart`).

## Настройки (`features/settings`)

Раздел «Больше»: системные настройки (тема/язык/формат времени — применяются мгновенно),
редактор активностей (создание/правка, FontAwesome-иконка, цвет, режим, тип Помодоро;
системные активности нельзя удалить, только архивировать), настройки Помодоро/воды/
расписания (сохранение = новая версия настроек, не перезапись).

`WaterSettingsPage` — экран настроек воды с собственным экранным кубитом
(`WaterSettingsCubit`/`WaterSettingsState`, `features/water/presentation/cubit/`): страница
не обращается к `WaterRepository` напрямую, только через кубит (общее правило — presentation
никогда не держит репозиторий, только Bloc/Cubit). Цель/режимы напоминаний/флаги туалета
редактируются локально в состоянии кубита и сохраняются явной кнопкой «Save»
(`WaterSettingsCubit.save`); время напоминаний и быстрые кнопки — персистятся сразу при
изменении, без отдельного сохранения.

Быстрые кнопки (`WaterQuickButtonEntity`): на экране настроек видны **все** кнопки, включая
выключенные (`isActive = false`) — переключатель просто меняет их активность, не скрывает
из списка. Для этого `WaterRepository`/`WaterDao` даёт два разных стрима:
`watchQuickButtons()` (только активные — для long-press списка на HUD) и
`watchAllQuickButtons()` (все — для этого экрана).

`WaterQuickButtonEntity.label` — свободный текст (`WaterQuickButtons.label`, всегда был
`TextColumn`, миграция не понадобилась): 5 предустановленных значений (`water/tea/coffee/
milk/bottle`, имя `DrinkType`) переводятся через `localizedDrinkLabel`
(`shared/widgets/drink_localization.dart`), а любое другое сохранённое значение — имя,
введённое пользователем — показывается как есть, без перевода (та же схема, что и
переименованные системные активности, FR-042, `localizedActionName`).

Добавление/редактирование/удаление — одна и та же модальная форма (`_QuickButtonForm`):
`+` рядом с заголовком «Кнопки быстрого добавления» открывает её пустой (создание), тап по
самой кнопке в списке — с уже заполненными полями (редактирование) и кнопкой «Удалить».
Поля формы: свободное текстовое название, иконка через `IconPickerDialog` (тот же пикер, что
и у активностей), объём в мл. Переключатель активности вынесен отдельно (`Switch` в трейлинге
плитки списка) — тап по плитке не по нему открывает форму, не переключает активность.

Порядок кнопок — drag-and-drop (`ReorderableListView.builder`, единственный другой экран с
этим паттерном — редактор активностей), персистится через `sortOrder`:
`WaterDao.reorderQuickButtons`/`WaterSettingsCubit.reorderQuickButtons` перезаписывают
`sortOrder` каждой кнопки на её новый индекс в списке — тот же приём, что
`ActionDao.reorder` для активностей.

## Онбординг (`features/onboarding`)

Пропускаемый первый запуск: описание функций, опциональный ввод имени, запрос разрешения на
уведомления с корректной обработкой отказа. После `skip` приложение полностью работоспособно
с 12 предустановленными системными активностями; повторный запуск онбординг не показывает.
