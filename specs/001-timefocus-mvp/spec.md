# Feature Specification: TimeFocus MVP (Фаза 1)

**Feature Branch**: `001-timefocus-mvp`

**Created**: 2026-07-18

**Status**: Draft

**Input**: User description: "Мобильное приложение TimeFocus по PRD v1.0 (.project/TimeFocus_PRD.docx): трекер времени + Помодоро + трекер воды + расписание дня + единая система уведомлений, offline-first, без конфликтов между модулями."

## Clarifications

### Session 2026-07-18

- Q: Как учитывать активность, которая идёт через полночь? → A: Целиком в день старта —
  интервал не разрезается, сессия и вся её длительность относятся к дате начала.
- Q: Что происходит с Помодоро-интервалом при паузе Помодоро-активности? → A: Помодоро
  прерывается (статус «interrupted»); активность остаётся на паузе с сохранением времени,
  после возобновления начинается новый Помодоро-интервал.
- Q: Что входит в «суммарное рабочее время» в шапке истории? → A: Суммарное время всех
  активностей за период, кроме активности «Сон».
- Q: Что происходит в точное время строгого события? → A: Помодоро принудительно прерывается
  и приходит уведомление; связанная активность стартует только по тапу на уведомление
  (авто-старта нет).
- Q: Как обрабатывать пересечение интервалов одной активности при редактировании истории? →
  A: Предупредить о пересечении, но разрешить сохранение.
- Q: Что останавливает pauseOthers-активность? → A: Текущий рабочий интервал Помодоро →
  «interrupted», а все остальные активные активности ставятся на паузу и автоматически
  возобновляются после остановки pauseOthers-активности.
- Q: Туалет (pauseOthers) во время перерыва Помодоро? → A: Перерыв и цикл не затрагиваются —
  break-активность продолжается, туалет трекается параллельно; прерывание действует только
  во время рабочего интервала.
- Q: Что со счётчиком цикла после прерывания интервала? → A: Сохраняется — прерванный
  интервал не засчитывается, новый стартует с тем же номером; сброс цикла только по «Стоп»
  активности.
- Q: Как доставлять отложенные уведомления после конца Помодоро/мьюта? → A: Все накопленные,
  по очереди.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Трекер активностей (Priority: P1)

Пользователь открывает экран «Трекер», видит сетку иконок своих активностей и одним нажатием
запускает учёт времени. Запущенные активности отображаются карточками со счётчиком текущего
интервала и накопленным временем за день. Пользователь может поставить активность на паузу,
возобновить или остановить её. Несколько обычных активностей могут идти параллельно.
Активности могут объединяться в группы: нажатие на иконку группы перестраивает сетку in-place,
показывая активности группы и кнопку «Назад».

**Why this priority**: это ядро продукта — без учёта времени остальные модули (Помодоро, история,
расписание) не имеют данных для работы. Одна эта история — уже минимально полезный продукт.

**Independent Test**: запустить активность из сетки, убедиться, что таймер идёт (время =
текущий момент − момент старта + накопленное), поставить на паузу, возобновить, остановить —
интервалы сохранены и видны при повторном открытии приложения.

**Acceptance Scenarios**:

1. **Given** экран «Трекер» с сеткой активностей, **When** пользователь нажимает иконку
   активности, **Then** активность мгновенно стартует и появляется карточкой вверху списка
   запущенных с тикающим таймером.
2. **Given** запущенная активность, **When** пользователь нажимает «Пауза», **Then** таймер
   останавливается, карточка опускается в конец списка активных, накопленное время сохраняется.
3. **Given** активность на паузе, **When** пользователь нажимает «Старт», **Then** отсчёт
   продолжается с накопленного времени.
4. **Given** запущенная активность, **When** пользователь нажимает «Стоп», **Then** карточка
   исчезает с экрана, сессия с интервалами уходит в историю.
5. **Given** две обычные активности (режим «nothing», без pauseOthers), **When** обе запущены,
   **Then** обе идут параллельно, не влияя друг на друга.
6. **Given** приложение закрыто при запущенной активности, **When** пользователь открывает его
   через N минут, **Then** таймер показывает корректное время (вычислено от момента старта,
   без фонового процесса).
7. **Given** сетка с группой активностей, **When** пользователь нажимает иконку группы,
   **Then** сетка показывает активности группы и кнопку «Назад» без перехода на другой экран.

### User Story 2 - Помодоро-сессии и переходы (Priority: P2)

Пользователь настраивает активность «Работа» как Помодоро-активность (тип short/normal/long) и
связывает её с активностью-перерывом. При старте идёт рабочий интервал; по его истечении
приходит уведомление, происходит штатный переход в перерыв (по настройке «после интервала»:
ничего / авто-перерыв / предложить / повторить / авто-возврат). После каждых N рабочих
интервалов — длинный перерыв. Запуск прерывающей активности (туалет, еда, спорт, другой
Помодоро, ручной перерыв) прерывает текущий Помодоро со статусом «interrupted».

**Why this priority**: Помодоро — главная «фишка» гармонизации: единая матрица переходов
предотвращает конфликты между работой и потребностями. Строится поверх P1.

**Independent Test**: запустить Помодоро-активность, дождаться конца рабочего интервала —
получить уведомление и штатный переход в перерыв; запустить «Туалет» во время работы —
Помодоро прерван, показано пояснение.

**Acceptance Scenarios**:

1. **Given** Помодоро-активность запущена, **When** рабочий интервал истёк, **Then** приходит
   уведомление «pomodoroFinished», сессия помечается «completed», выполняется действие из
   настройки «после интервала».
2. **Given** активен Помодоро, **When** пользователь запускает активность с pauseOthers
   (туалет/еда/спорт), **Then** текущая активность останавливается, Помодоро → «interrupted».
3. **Given** активен Помодоро активности А, **When** пользователь запускает Помодоро-активность Б,
   **Then** показывается предупреждение о прерывании; при подтверждении А останавливается,
   её Помодоро → «interrupted».
4. **Given** рабочий интервал завершился системно, **When** стартует связанная break-активность,
   **Then** Помодоро НЕ прерывается — это единственный штатный случай перехода в перерыв.
5. **Given** завершено N рабочих интервалов (по настройке, по умолчанию 4), **When** наступает
   перерыв, **Then** он длинный, счётчик цикла отображается на карточке (например «2/4»).
6. **Given** перерыв истёк, **When** приходит уведомление «breakFinished», **Then** пользователь
   может вернуться к работе (тап по уведомлению) или продлить перерыв.
7. **Given** Помодоро-интервал идёт, **When** пользователь нажимает «Пропустить», **Then**
   интервал помечается «skipped» и цикл продолжается.
8. **Given** Помодоро-интервал идёт, **When** пользователь ставит активность на паузу,
   **Then** интервал помечается «interrupted», уведомление отменяется; после возобновления
   активности стартует новый Помодоро-интервал.

### User Story 3 - Трекер воды и HUD-панель (Priority: P3)

Над нижней навигацией постоянно видна HUD-панель: полоска воды с делениями-порциями, оранжевой
меткой текущей нормы и красной зоной дефицита. Нажатие на кнопку стакана добавляет порцию
(200 мл), удержание открывает список напитков (вода/чай/кофе/молоко/бутылка) с разными
объёмами. Приложение напоминает пить воду в одном из двух режимов: «интервал от последнего
питья» или «по расписанию». Справа в HUD — контекстная иконка (туалет/еда/спорт/сон) по
приоритету и времени суток.

**Why this priority**: гидратация — второй по важности модуль здоровья; HUD — точка гармонизации
(вода/туалет/еда) поверх трекера и Помодоро.

**Independent Test**: нажать стакан — лог воды +200 мл и полоска обновилась; удержать — выбрать
«Бутылка 500 мл»; включить режим «интервал» — уведомление приходит через N минут после питья.

**Acceptance Scenarios**:

1. **Given** HUD-панель видна, **When** пользователь нажимает стакан, **Then** в журнал
   добавляется 200 мл и полоска воды обновляется мгновенно.
2. **Given** HUD-панель видна, **When** пользователь удерживает стакан, **Then** открывается
   список напитков с редактируемыми объёмами; выбор добавляет соответствующий объём.
3. **Given** режим напоминаний «интервал», **When** пользователь пьёт воду, **Then** следующее
   напоминание переносится на «время питья + N минут».
4. **Given** режим «по расписанию», **When** пользователь пьёт воду, **Then** расписание
   напоминаний не меняется; напоминания в окне ±15 минут от приёма пищи автоматически
   пропускаются.
5. **Given** выпито меньше нормы к текущему времени, **When** приходит напоминание о воде,
   **Then** в нём рекомендуется число стаканов = округление вверх (дефицит / 200), но не более 4.
6. **Given** активен Помодоро, **When** наступает время напоминания о воде, **Then** уведомление
   откладывается до ближайшего перерыва, а иконка стакана в HUD мигает.
7. **Given** включена настройка «туалет после воды» и нет других контекстных виджетов,
   **When** пользователь фиксирует питьё, **Then** в HUD появляется пульсирующая иконка туалета;
   нажатие запускает активность «Туалет» (Помодоро прерывается).
8. **Given** несколько контекстных триггеров одновременно, **When** HUD выбирает иконку,
   **Then** применяется приоритет: Туалет > Еда > Спорт > Сон > пусто.

### User Story 4 - Расписание дня и строгие/гибкие события (Priority: P4)

Пользователь настраивает распорядок дня (подъём, завтрак, работа, обед, спорт, ужин, сон) —
отдельно для будней и выходных. Экран «Распорядок дня» показывает вертикальный таймлайн:
плановые блоки, реальные интервалы трекера поверх плана, точки приёма воды и активные
напоминания; событие можно добавить прямо с таймлайна. Гибкое событие откладывает уведомление
до конца Помодоро; строгое — предупреждает заранее, если Помодоро не успевает, и принудительно
прерывает его в точное время.

**Why this priority**: расписание связывает все модули в единый день, но полезно только когда
уже есть трекер, Помодоро и вода.

**Independent Test**: задать строгий обед на 13:00, запустить Помодоро в 12:45 (интервал
25 минут) — немедленно получить предупреждение «не успеет»; в 13:00 Помодоро прерван и
предложен старт активности «Приём пищи».

**Acceptance Scenarios**:

1. **Given** настроенное расписание дня, **When** пользователь открывает «Распорядок дня»,
   **Then** видит таймлайн: плановые блоки, фактические интервалы, точки воды, напоминания.
2. **Given** гибкое событие «обед», **When** его время наступает при активном Помодоро,
   **Then** уведомление откладывается до конца Помодоро, иконка еды в HUD пульсирует.
3. **Given** строгое событие, **When** пользователь стартует Помодоро, который не успеет
   завершиться до события, **Then** немедленно приходит предупреждение.
4. **Given** строгое событие наступило при активном Помодоро, **When** точное время достигнуто,
   **Then** Помодоро принудительно прерывается и приходит уведомление события.
5. **Given** таймлайн открыт, **When** пользователь добавляет событие вручную, **Then** оно
   появляется на таймлайне и участвует в планировании уведомлений.

### User Story 5 - Уведомления и холодный старт (Priority: P5)

Пользователь получает локальные уведомления всех типов (конец Помодоро, конец перерыва, еда
гибкая/строгая, предупреждение, вода, туалет, сон, продление перерыва). Тап по уведомлению
открывает приложение — даже из полностью закрытого состояния — и выполняет нужный сценарий
(старт перерыва, возврат к работе, запуск активности события). Активность «Сон», «Медитация»,
«Молитва» глушат уведомления; есть и глобальный мьют.

**Why this priority**: уведомления пронизывают P2–P4; выделены отдельно, потому что сценарий
«тап → холодный старт → действие» и правила откладывания тестируются самостоятельно.

**Independent Test**: получить уведомление «конец рабочего интервала» при закрытом приложении,
тапнуть — приложение открылось и стартовал перерыв связанной активности.

**Acceptance Scenarios**:

1. **Given** приложение полностью закрыто, **When** пользователь тапает по уведомлению
   «pomodoroFinished», **Then** приложение открывается и выполняет переход в перерыв —
   вся необходимая информация содержится в самом уведомлении.
2. **Given** первый запуск после установки на современной версии ОС, **When** пользователь
   доходит до включения напоминаний, **Then** запрашиваются необходимые системные разрешения,
   отказ обрабатывается без сбоя (напоминания выключены, функции трекинга доступны).
3. **Given** активность «Сон» запущена, **When** наступает время любого напоминания,
   **Then** уведомление не показывается (мьют).
4. **Given** уведомление «breakFinished» с кнопкой продления, **When** пользователь выбирает
   «продлить», **Then** перерыв продлевается и планируется новое уведомление.
5. **Given** за 30 минут до времени сна из расписания, **When** нет активного Помодоро,
   **Then** приходит напоминание «пора спать»; при активном Помодоро — откладывается до его конца.

### User Story 6 - История и отчёты (Priority: P6)

Пользователь открывает «Историю», переключает режимы (Интервалы / Итого по активностям /
Статистика) и периоды (День / Неделя / Месяц / Год), листает периоды стрелками, возвращается
к сегодняшнему дню одной кнопкой. Шапка показывает суммарное рабочее время, число завершённых
и прерванных Помодоро, выпитую воду против цели. Записи можно редактировать: менять активность
интервала, статус, комментарий, время начала/конца (с быстрыми кнопками −5/−1/+1/+5/сейчас),
добавлять интервалы вручную и удалять. Раздел «Отчеты» даёт готовые срезы (сегодня, вчера,
эта/прошлая неделя, этот/прошлый месяц, последние 30 дней).

**Why this priority**: ценность истории появляется после накопления данных из P1–P3.

**Independent Test**: остановить активность, найти её в «Истории» в режиме «Интервалы»,
отредактировать время окончания, убедиться, что итоги пересчитались.

**Acceptance Scenarios**:

1. **Given** накопленные записи, **When** пользователь переключает период на «Неделя»,
   **Then** список и шапка (работа, Помодоро, вода) пересчитываются для недели.
2. **Given** режим «Интервалы», **When** пользователь нажимает запись, **Then** открывается
   экран сессии: смена активности, статус, комментарий, список интервалов, добавление интервала.
3. **Given** экран редактирования интервала, **When** пользователь применяет быстрые кнопки
   времени, **Then** длительность отображается и сохраняется корректно; конец не может быть
   раньше начала.
4. **Given** экран сессии, **When** пользователь удаляет сессию, **Then** удаляются все её
   интервалы и статистика пересчитывается.

### User Story 7 - Настройки и персонализация (Priority: P7)

Через раздел «Больше» пользователь настраивает: тему (тёмная/светлая/системная), язык
(en/uk/ru/системный), формат времени (12/24, с секундами и без); активности (создание с иконкой,
цветом, названием, типом; редактирование системных; архивация; размер сетки 1–5 × 1–5);
параметры Помодоро (времена трёх типов, число циклов до длинного перерыва, действие после
интервала, звук/вибрация); параметры воды (цель по весу или вручную, режим напоминаний, окно
напоминаний, напитки, настройки туалета); напоминания (вкл/выкл, разрешения). Смена темы и
языка применяется мгновенно, без перезапуска.

**Why this priority**: приложение работает на разумных значениях по умолчанию; настройки
расширяют, но не блокируют основные сценарии.

**Independent Test**: сменить язык на украинский — интерфейс переключился сразу; создать
активность с иконкой и цветом — она появилась в сетке; изменить время «normal»-Помодоро —
следующая сессия использует новое значение.

**Acceptance Scenarios**:

1. **Given** экран системных настроек, **When** пользователь меняет тему или язык,
   **Then** интерфейс обновляется мгновенно и выбор сохраняется между запусками.
2. **Given** экран настроек активностей, **When** пользователь создаёт активность (название,
   иконка, цвет, тип), **Then** она появляется в сетке трекера.
3. **Given** системная активность, **When** пользователь пытается её удалить, **Then** удаление
   недоступно; доступны переименование и настройка.
4. **Given** настройки Помодоро, **When** пользователь меняет длительности типа,
   **Then** новые значения применяются ко всем активностям этого типа со следующей сессии,
   а прошлая статистика не искажается.
5. **Given** настройки воды в режиме «по весу», **When** пользователь вводит вес,
   **Then** дневная цель пересчитывается; смена цели не переписывает нормы прошлых дней.

### User Story 8 - Онбординг и первый запуск (Priority: P8)

При первом запуске пользователь видит короткий, полностью пропускаемый онбординг: зачем
приложение и какие есть функции, ввод имени (опционально). Без регистрации, email и внешних
аккаунтов. После онбординга доступно всё: предустановленные системные активности, расписание
по умолчанию, разумные нормы воды.

**Why this priority**: важен для первого впечатления, но не блокирует ни одну функцию —
онбординг можно пропустить целиком.

**Independent Test**: установить приложение, пропустить онбординг — экран «Трекер» работает
со всеми системными активностями; повторный запуск онбординг не показывает.

**Acceptance Scenarios**:

1. **Given** первый запуск, **When** пользователь пропускает онбординг, **Then** попадает на
   «Трекер» с предустановленными активностями и настройками по умолчанию.
2. **Given** онбординг пройден или пропущен, **When** приложение перезапускается,
   **Then** онбординг больше не показывается.

### Edge Cases

- Приложение убито системой при активном таймере → при следующем запуске время корректно
  вычисляется от сохранённого момента старта; запланированные уведомления не теряются.
- Смена системного времени/часового пояса при активном таймере → интервалы не становятся
  отрицательными; уведомления перепланируются по локальному времени.
- Активность идёт через полночь → интервал не разрезается: сессия и вся её длительность
  учитываются в дне старта; дневная норма воды фиксируется на дату по действовавшей в тот
  день цели.
- Пользователь отклонил разрешение на уведомления → все напоминания выключаются без сбоев,
  остальные функции работают; в настройках есть переход в системные настройки разрешений.
- Строгое событие наступает во время перерыва Помодоро (не рабочего интервала) → событие
  выполняется, перерыв завершается штатно.
- Строгое событие совпадает с моментом завершения рабочего интервала → интервал засчитывается
  как «completed», затем обрабатывается событие (прерывания нет — интервал уже завершён).
- Родительская Помодоро-активность остановлена во время перерыва → перерыв завершается
  штатно, уведомление «конец перерыва» не запускает остановленную активность автоматически,
  тап открывает трекер.
- Одновременный тап по двум иконкам сетки / двойной тап → активность стартует один раз.
- Удаление пользовательской активности, на которую ссылаются история, расписание или связка
  «перерыв для» → история сохраняется, ссылки расписания корректно очищаются, связка
  break-активности сбрасывается на «просто пауза».
- Редактирование интервала истории с пересечением с другим интервалом той же активности →
  система показывает предупреждение, но разрешает сохранение; запрещён только диапазон,
  где конец раньше начала.
- 0 запущенных активностей → список пуст, сетка занимает максимум места; 10+ параллельных
  активностей → список прокручивается, производительность не деградирует.
- Все напитки отключены в настройках → кнопка стакана продолжает работать с базовой порцией.

## Requirements *(mandatory)*

### Functional Requirements

**Трекер активностей**

- **FR-001**: Система MUST позволять мгновенный запуск активности одним нажатием на иконку в
  сетке и отображать её карточкой в списке запущенных.
- **FR-002**: Система MUST поддерживать три состояния активности — активна, на паузе,
  остановлена — с сохранением накопленного времени между состояниями и перезапусками приложения.
- **FR-003**: Система MUST вычислять прошедшее время как разницу «сейчас − момент старта» плюс
  накопленное время, без фоновых процессов.
- **FR-004**: Система MUST поддерживать параллельный трекинг любого числа обычных активностей
  (режим «nothing» без pauseOthers).
- **FR-005**: Система MUST сортировать карточки: активные по времени запуска (новейшая вверху),
  затем активности на паузе по времени остановки; остановленные скрываются.
- **FR-006**: Карточка MUST показывать иконку, название, таймер текущего интервала, общее время
  за день, индикатор Помодоро/паузы с типом и номером цикла (например «2/4»), кнопки
  Пауза/Старт и Стоп. На паузе таймер интервала показывает замороженное значение на момент
  паузы.
- **FR-007**: Система MUST поддерживать группы активностей: нажатие на группу перестраивает
  сетку in-place (активности группы + кнопка «Назад»), без отдельного экрана.
- **FR-008**: Система MUST предоставлять предустановленные системные активности (Работа,
  Перерыв, Отдых, Сон, Туалет, Приём пищи, Спорт, Разминка, Прогулка, Медитация, Молитва,
  Лекарства) с заданными свойствами (pauseOthers, длительность по умолчанию, мьют
  уведомлений); их нельзя удалить, но можно переименовать и настроить.
- **FR-009**: Сетка активностей MUST настраиваться по размеру (колонки 1–5, ряды 1–5); при
  числе активностей больше вместимости сетка прокручивается.

**Переходы и прерывания**

- **FR-010**: Система MUST применять единую матрицу переходов: активность прерывает текущий
  Помодоро тогда и только тогда, когда она Помодоро-активность, ручной перерыв или имеет
  pauseOthers; системное завершение рабочего интервала — единственный случай, когда переход
  в перерыв НЕ прерывает Помодоро.
- **FR-010a**: При старте pauseOthers-активности: текущий рабочий интервал Помодоро →
  «interrupted»; все прочие активные активности ставятся на паузу (система помечает их как
  приостановленные автоматически) и MUST автоматически возобновляться после остановки
  pauseOthers-активности.
- **FR-010b**: Во время перерыва Помодоро pauseOthers-активность (например, «Туалет» по
  виджету HUD) MUST NOT затрагивать break-активность и счётчик цикла — она трекается
  параллельно; прерывание распространяется только на рабочий интервал.
- **FR-011**: При запуске второй Помодоро-активности система MUST предупредить пользователя о
  прерывании текущей и продолжить только после подтверждения; при отказе ни одна из
  активностей не меняет состояния.
- **FR-012**: При прерывании Помодоро система MUST показать пользователю понятное объяснение
  (какая активность прервана и чем).

**Помодоро**

- **FR-013**: Система MUST поддерживать три глобальных типа Помодоро (short 15 мин, normal
  25 мин, long 45 мин по умолчанию) с настраиваемыми временами работы и перерывов.
- **FR-014**: Длинный перерыв MUST наступать после настраиваемого числа рабочих интервалов
  (по умолчанию 4, диапазон 3–5). Прерванный (interrupted) интервал не засчитывается в цикл:
  следующий интервал стартует с тем же номером; счётчик цикла сбрасывается только при «Стоп»
  активности.
- **FR-015**: Для активности MUST настраиваться: включение Помодоро (выбор типа или «выключен»)
  и связанная активность-перерыв; отсутствие связки означает паузу таймера без переключения.
- **FR-016**: Активность-перерыв MUST выполнять только обратный отсчёт перерыва (без
  собственного Помодоро) и MAY быть перерывом для нескольких Помодоро-активностей.
- **FR-017**: Система MUST фиксировать каждый интервал Помодоро со статусом: активен, завершён
  (только системно), прерван (пользователем/системой), пропущен — и сохранять снимок настроек,
  действовавших на момент сессии.
- **FR-018**: Система MUST поддерживать настраиваемое действие после рабочего интервала:
  ничего / авто-старт перерыва / предложить перерыв / повторить интервал / авто-возврат к
  работе. Если приложение закрыто в момент срабатывания, авто-действие выполняется при
  следующем открытии (по тапу на уведомление или запуску), время старта — момент открытия.
- **FR-018a**: Пауза Помодоро-активности MUST прерывать текущий Помодоро-интервал (статус
  «interrupted», запланированное уведомление отменяется); активность остаётся на паузе с
  сохранением накопленного времени, при возобновлении начинается новый Помодоро-интервал.

**Вода и HUD**

- **FR-019**: HUD-панель MUST постоянно отображаться над нижней навигацией на вкладках оболочки
  и содержать полоску воды (деления-порции, метка текущей нормы, зона дефицита), кнопку стакана
  и одну контекстную иконку.
- **FR-020**: Нажатие на стакан MUST добавлять порцию (по умолчанию 200 мл, настраивается
  100–500 мл); удержание MUST открывать редактируемый список напитков (по умолчанию: вода
  200 мл, чай 150 мл, кофе 100 мл, молоко 200 мл, бутылка 500 мл).
- **FR-021**: Дневная цель воды MUST рассчитываться по весу пользователя или задаваться
  вручную, с учётом дополнительной нагрузки; норма «к текущему времени» распределяется
  равномерно от подъёма до сна.
- **FR-022**: Система MUST хранить историю дневных целей, чтобы статистика прошлых дней
  считалась по действовавшей в тот день норме.
- **FR-023**: Режим «интервал»: следующее напоминание MUST планироваться как «последнее питьё +
  N минут» и пересоздаваться после каждого питья.
- **FR-024**: Режим «по расписанию»: все напоминания дня MUST планироваться сразу; питьё не
  сдвигает расписание; напоминания в окне ±15 минут от приёма пищи автоматически пропускаются.
- **FR-025**: При активном Помодоро напоминание о воде MUST откладываться до ближайшего
  перерыва, при этом иконка стакана в HUD мигает. Если у Помодоро-активности нет связанного
  перерыва, «перерывом» считается завершение или прерывание рабочего интервала. Напоминание,
  не доставленное до конца дневного окна напоминаний (время сна), отбрасывается.
- **FR-026**: В напоминании о дефиците система MUST рекомендовать число стаканов: округление
  вверх (дефицит / объём порции), но не более 4.
- **FR-027**: Контекстная иконка HUD MUST выбираться по приоритету Туалет > Еда > Спорт > Сон >
  пусто; иконка еды меняется по времени суток (завтрак/обед/ужин/перекус).
- **FR-028**: Виджет туалета MUST появляться только по двум триггерам — после питья (настройка
  «туалет после воды») и при старте перерыва Помодоро (настройка «туалет на перерыве»); обе
  настройки выключены → виджет и уведомления туалета полностью отсутствуют. Нажатие на
  пульсирующий виджет запускает активность «Туалет».

**Расписание дня**

- **FR-029**: Система MUST поддерживать настраиваемое расписание дня (подъём, приёмы пищи,
  работа, спорт, сон, произвольные события) с отдельными наборами для будней и выходных.
- **FR-030**: Экран «Распорядок дня» MUST отображать вертикальный таймлайн текущего дня:
  плановые блоки, фактические интервалы трекера поверх плана, точки приёма воды, активные
  напоминания — и позволять добавлять события вручную с таймлайна. Параллельные фактические
  интервалы и события с одинаковым временем отображаются без взаимного перекрытия
  (отдельные дорожки или смещение).
- **FR-031**: Гибкое событие MUST откладывать уведомление до конца текущего Помодоро (иконка в
  HUD пульсирует); строгое событие MUST принудительно прерывать Помодоро в точное время и
  отправлять уведомление; связанная активность события стартует только по тапу пользователя
  на уведомление — автоматический запуск активности не выполняется.
- **FR-032**: При старте каждого Помодоро система MUST проверять строгие события дня и
  немедленно предупреждать, если интервал не успеет завершиться до события; предупреждения
  пересчитываются при каждом старте активности.
- **FR-033**: Архитектура MUST предусматривать точку расширения для интеграции с системным
  календарём (реализация — Фаза 2).

**Уведомления**

- **FR-034**: Система MUST поддерживать типы уведомлений: конец рабочего интервала, конец
  перерыва, гибкая еда, строгое событие, предупреждение о строгом событии, вода, туалет,
  сон (за 30 минут), продление перерыва — с правилами прерывания/откладывания из PRD.
  Продление перерыва: на настраиваемое число минут (по умолчанию 5), допускается многократно.
- **FR-034a**: Отложенные из-за Помодоро или мьюта уведомления MUST доставляться все,
  по очереди, после окончания Помодоро/мьюта. В планировщике MUST NOT существовать более
  одного недоставленного уведомления одного типа с одинаковым контекстом (детерминированный
  ключ из payload: activityId для Помодоро/перерыва, scheduleEventId для событий; для воды —
  сам тип) — новое заменяет старое (нет дублей, SC-007).
- **FR-035**: Все уведомления MUST быть локальными, планироваться заранее и содержать в payload
  всё необходимое для выполнения действия при холодном старте; тап открывает приложение и
  запускает нужный сценарий (deep-link обязателен). Тап по устаревшему уведомлению (активность
  удалена/остановлена, событие прошло) MUST открывать приложение без сбоя и без выполнения
  действия.
- **FR-035a**: Запланированные уведомления MUST переживать перезагрузку устройства
  (восстановление средствами ОС/механизма уведомлений); дополнительно при каждом старте
  приложения система пересоздаёт актуальные уведомления из своей базы.
- **FR-036**: Система MUST запрашивать необходимые разрешения на уведомления и точные
  будильники в зависимости от версии ОС и корректно работать при отказе (напоминания
  отключены, трекинг доступен, доступен переход в системные настройки). При выданных
  уведомлениях, но отклонённых точных будильниках допускается неточная доставка (режим
  «примерно в это время») без потери функциональности.
- **FR-037**: Активности с мьютом (Сон, Медитация, Молитва) MUST глушить все напоминания на
  время своей работы; также MUST существовать глобальный выключатель уведомлений.

**История и отчёты**

- **FR-038**: Экран «История» MUST поддерживать режимы «Интервалы», «Итого по активностям»,
  «Статистика» и периоды День/Неделя/Месяц/Год с навигацией стрелками и возвратом к «сегодня».
- **FR-039**: Шапка истории MUST показывать за период: суммарное затреканное время (все
  активности, кроме «Сна»), число завершённых и прерванных Помодоро, воду «выпито / цель».
- **FR-040**: Пользователь MUST иметь возможность редактировать историю: менять активность
  сессии, статус, комментарий; редактировать, добавлять и удалять интервалы (быстрые кнопки
  времени: сейчас, −5, −1, +1, +5 минут); удалять сессию со всеми интервалами. Конец интервала
  не может быть раньше начала; при пересечении с другим интервалом той же активности система
  предупреждает, но позволяет сохранить.
- **FR-041**: Раздел «Отчеты» MUST предоставлять срезы: сегодня, вчера, эта неделя, прошлая
  неделя, этот месяц, прошлый месяц, последние 30 дней.

**Настройки, локализация, онбординг**

- **FR-042**: Система MUST поддерживать темы тёмная/светлая/системная и языки en/uk/ru/системный
  с мгновенным применением без перезапуска; формат времени 12/24 и краткий формат без секунд
  применяются во всех местах отображения времени. Дефолтные названия системных активностей и
  напитков локализуются при смене языка; переименованные пользователем — сохраняются как есть
  и не переводятся.
- **FR-046**: Все экраны MUST иметь дружественные пустые состояния (история без записей,
  таймлайн без событий, пустая сетка); ошибки операций с данными показываются единообразно:
  тост для глобальных сбоев, inline-сообщение у полей форм.
- **FR-047**: Доступность: интерактивные элементы сетки и HUD MUST иметь тап-цель не менее
  48×48 dp и текстовые метки для screen reader; пульсация/мигание индикаторов отключаются при
  системной настройке уменьшения анимации; полоска воды и красная зона различимы не только
  цветом.
- **FR-043**: Пользователь MUST иметь возможность создавать активности (название, описание,
  иконка, цвет, режим, тип Помодоро, связка-перерыв, длительность по умолчанию), редактировать
  и архивировать их (архив скрывает из сетки, не удаляя данные).
- **FR-044**: Онбординг MUST показываться только при первом запуске, быть полностью
  пропускаемым и не требовать регистрации; приложение MUST быть полностью работоспособно
  сразу после установки (offline, значения по умолчанию, системные активности).
- **FR-045**: Раздел «Учетная запись» MUST присутствовать как задел: создание учётной записи,
  синхронизация и резервное копирование заявлены в интерфейсе, но их реализация — вне объёма
  MVP; отсутствие учётной записи не ограничивает ни одну функцию.

### Key Entities

- **Активность (ActionName)**: справочник дел пользователя — название, иконка, цвет, режим
  (обычная / Помодоро / перерыв-для), тип Помодоро, связка с перерывом, pauseOthers,
  длительность по умолчанию, признак системной, группа, приоритет в HUD, архив.
- **Запущенная активность (ActionRunning)**: текущее выполнение — статус (активна/пауза/стоп),
  момент старта, момент паузы, накопленные секунды; ссылается на активность и сессию истории.
- **Сессия истории (ActionHistory)**: одна запись на активность и день — дата (всегда дата
  старта сессии, интервалы через полночь не разрезаются), комментарий;
  содержит **интервалы (ActionHistoryInterval)** — пары «начало–конец».
- **Помодоро-сессия (PomodoroSession)**: рабочий интервал или перерыв — тип, план/факт времени,
  статус (активен/завершён/прерван/пропущен), номер в цикле, снимок настроек.
- **Настройки Помодоро (PomodoroSettings)**: времена трёх типов, перерывы, циклов до длинного
  перерыва, действие после интервала, звук/вибрация; версионируются (история изменений).
- **Событие расписания (ScheduleEvent)**: тип (подъём/еда/работа/спорт/сон/своё), подтип еды,
  время и длительность, строгое/гибкое, минуты предупреждения, связанная активность, вкл/выкл.
- **Журнал воды (WaterLog)** и **дневные цели (DailyWaterGoal)**: приёмы воды с объёмом и
  временем; зафиксированная норма на каждую дату.
- **Настройки воды (WaterSettings)**: цель (по весу/вручную), режим напоминаний, интервал,
  времена расписания (**WaterReminderTime**), напитки (**WaterQuickButton**), флаги туалета.
- **Уведомление (Notification)**: тип, время срабатывания, payload для холодного старта,
  признак доставки.
- **Настройки пользователя (UserSettings)**: имя, тема, язык, формат времени, размер сетки,
  глобальный мьют, флаг онбординга, состояние разрешений.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Старт активности из сетки — не более 1 нажатия; карточка с идущим таймером
  появляется менее чем за 1 секунду.
- **SC-002**: Показания таймера после перезапуска приложения или 8 часов простоя расходятся с
  реальным временем не более чем на 1 секунду.
- **SC-003**: 100% уведомлений, по которым тапнул пользователь, приводят к корректному сценарию
  (в том числе при холодном старте приложения).
- **SC-004**: Полный цикл Помодоро (работа → уведомление → перерыв → возврат к работе)
  проходит без единого ручного действия при настройках авто-старта и открытом приложении;
  при закрытом приложении авто-действие выполняется при первом открытии (FR-018).
- **SC-005**: Фиксация питья воды — 1 нажатие для стандартной порции, не более 2 действий для
  любого напитка из списка.
- **SC-006**: Приложение полностью функционально без сети и без учётной записи: 100% сценариев
  P1–P8 выполняются offline.
- **SC-007**: Пользователь получает не более одного напоминания на событие: конфликтующие
  напоминания (вода во время Помодоро, еда рядом с водой) объединяются или откладываются по
  правилам — дублирующихся уведомлений нет.
- **SC-008**: Смена темы или языка применяется без перезапуска с видимым обновлением не более
  чем за 1 секунду, корректно на всех экранах для en/uk/ru; «мгновенные» операции (лог воды,
  перестройка сетки, обновление HUD) дают видимый отклик не более чем за 200 мс.
- **SC-009**: Новый пользователь после пропуска онбординга запускает первую активность не более
  чем за 30 секунд без инструкций.
- **SC-010**: История корректно агрегирует данные за любой период (день/неделя/месяц/год);
  редактирование интервала немедленно пересчитывает итоги периода.

## Assumptions

- Объём — MVP Фаза 1 по PRD v1.0: интеграция с системным календарём, реальная синхронизация,
  резервное копирование и регистрация — вне объёма (закладываются только точки расширения и
  пункты интерфейса раздела «Учетная запись»).
- Платформы — Android и iOS (телефоны); планшетная раскладка не оптимизируется в MVP.
- «Молитва» входит в системные активности согласно PRD (в дополнение к списку из CLAUDE.md).
- Значения по умолчанию берутся из PRD: порция 200 мл, окно автопропуска еды ±15 мин,
  напоминание о сне за 30 мин, длительности системных активностей (туалет 3 мин, еда 20 мин,
  спорт 30 мин, разминка 5 мин, лекарства 2 мин).
- Точность таймера при открытом экране — визуальное обновление не реже 1 раза в секунду.
- Отчёты в MVP строятся на тех же данных, что и «История» (без экспорта в файлы).
- Первая настройка расписания предзаполняется типовым распорядком (подъём, завтрак, работа,
  обед, работа, спорт, ужин, сон), который пользователь может изменить.
